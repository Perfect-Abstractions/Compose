<DocSubtitle>
Implements ERC-20 token functionality within a diamond
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements core ERC-20 functions via external calls through the diamond proxy.
- Manages token state (totalSupply, balances, allowances) within diamond storage.
- Self-contained facet with no external dependencies or inheritance.
- Utilizes custom errors for gas-efficient error handling.
</Callout>

## Overview

This facet provides standard ERC-20 token operations, including transfers, approvals, and balance inquiries, routed through the diamond proxy. It accesses shared storage for token metadata and balances, enabling upgradeability and composability for token contracts within the diamond storage pattern.

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize token name, symbol, decimals, and total supply during diamond deployment.
- Enforce access control where necessary for functions like `approve` if specific roles are required.
- Ensure storage compatibility when upgrading the ERC20Facet to maintain state integrity.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
All state-changing functions (`approve`, `transfer`, `transferFrom`) emit events. Input validation is performed, reverting with custom errors such as `ERC20InsufficientBalance`, `ERC20InsufficientAllowance`, `ERC20InvalidSender`, `ERC20InvalidReceiver`, and `ERC20InvalidSpender` to prevent common ERC-20 vulnerabilities. Follow standard Solidity security practices for diamond interactions.
</Callout>
