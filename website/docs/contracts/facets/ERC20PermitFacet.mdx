Contract documentation for ERC20PermitFacet
</DocSubtitle>

<Callout type="info" title="Key Features">
- Implements EIP-2612 permit functionality for gas-efficient token approvals.
- Provides &#x60;nonces&#x60; to track and prevent replay attacks for permits.
- Exposes &#x60;DOMAIN_SEPARATOR&#x60; for correct signature verification against the specific diamond instance and chain ID.
</Callout>

## Overview

The ERC20PermitFacet enables EIP-2612 compliant ERC20 token approvals via off-chain signatures. It integrates with the diamond proxy to manage token permits, allowing users to grant allowances without direct on-chain transactions for each approval.

## Usage Example

<ExpandableCode language="solidity" maxLines={20} title="Example Implementation">
{`pragma solidity ^0.8.30;

import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import {IDiamondCut} from "../diamond/IDiamond.sol";
import {ERC20PermitFacet} from "./ERC20PermitFacet.sol";

contract Deployer {
    function deploy() external {
        // Assume diamond is deployed and facets are registered
        address diamond = address(0x123); // Replace with actual diamond address

        ERC20PermitFacet permitFacet = new ERC20PermitFacet();

        IDiamondCut(diamond).diamondCut(
            new IDiamondCut.FacetCut[](
                {facetAddress: address(permitFacet), action: IDiamondCut.FacetCutAction.ADD, selectors: IDiamondCut.getSelectors(permitFacet)}
            ),
            address(0),
            ""
        );

        // To use permit:
        // 1. Get nonce: ERC20PermitFacet(diamond).nonces(owner)
        // 2. Get domain separator: ERC20PermitFacet(diamond).DOMAIN_SEPARATOR()
        // 3. Construct permit data and sign off-chain
        // 4. Call permit:
        // ERC20PermitFacet(diamond).permit(owner, spender, value, deadline, v, r, s);
    }
}`}
</ExpandableCode>

## Best Practices

<Callout type="tip" title="Best Practice">
- Initialize the facet during diamond deployment or upgrade, ensuring the &#x60;DOMAIN_SEPARATOR&#x60; is correctly configured for the deployed chain.
- Use the &#x60;nonces&#x60; and &#x60;DOMAIN_SEPARATOR&#x60; functions to construct valid permit data off-chain before calling the &#x60;permit&#x60; function.
- Ensure the owner of the token correctly signs the permit message to prevent unauthorized approvals.
</Callout>

## Security Considerations

<Callout type="warning" title="Security">
The &#x60;permit&#x60; function relies on off-chain signatures. Ensure the signature verification process is robust and that the &#x60;owner&#x60; address signing the permit has sufficient balance and is the intended grantor of the allowance. The &#x60;nonces&#x60; mapping prevents replay attacks; it&#x27;s crucial that this mapping is managed correctly and incremented upon successful permit usage. The &#x60;DOMAIN_SEPARATOR&#x60; ensures that signatures are chain-specific, preventing cross-chain replay attacks.
</Callout>
